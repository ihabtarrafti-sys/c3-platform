The matcher was narrowed to `SELECT count(*)::int AS n` plus the prepared predicate (`webv0/packages/persistence/test/sweepCensusRace.test.ts:44-59`), then the same production neuters produced both RED failures quoted above. ## Cost envelopes

- **J′ permanent authority and cadence:** one small heap row plus its primary-key/index entry is retained per finalized tenant forever; tenant exits are rare owner ceremonies, so expected `N` is small, but there is no coded cardinality ceiling (`webv0/packages/persistence/migrations/0078_erased_tenant_prefix.sql:1-23`). Each boot/daily/owner pass scans all `N` permanent rows and performs at least two complete prefix listings per owned row; caught keys add deletes and repeated two-prefix listings until empty (`webv0/apps/api/src/erasureJanitor.ts:50-148`). The default and maximum configured interval is **86,400,000 ms**; tests may shrink it but configuration cannot exceed one day (`webv0/apps/api/src/env.ts:72-73`; `webv0/apps/api/test/env.test.ts:169-180`). Boot availability parks API listen for the full catch-up pass (`webv0/apps/api/src/server.ts:47-50`). Actual byte lifetime is conditional on API/timer execution and a successful storage/DB pass: an outage that starts before the next tick adds both the pre-outage interval remainder and the outage; an interval that overlaps an active pass may coalesce; failures add retry delay. Pass duration depends on `N`, storage pagination/key count, and provider response times and has no coded numeric ceiling (`webv0/apps/api/src/erasureJanitor.ts:218-268`; `webv0/apps/api/src/storage.ts:79-100`). Therefore this evidence makes **no unconditional numeric lifetime claim** and does not repeat the ruling/register's falsifiable `max(interval, API downtime) + pass duration` formula; that acceptance wall is filed in `C3-HARDEN-3.7-ESCALATION.md`. Permanent authority still has zero expiry dependence on provider publication time and retains every failed row for retry.
- **U4 checkout:** `createPersistence` defaults pool checkout/connection wait to **10,000 ms**, rejects non-positive/non-safe configured values, and the default is below the supported production request-deadline minimum of **30,000 ms** (`webv0/packages/persistence/src/stores.ts:18-54`; `webv0/apps/api/src/env.ts:61-69`). An already-aborted call refuses before queueing; a signal that fires while queued is rechecked as soon as checkout returns, and a saturated checkout independently fails by 10 seconds (`webv0/packages/persistence/src/tenantContext.ts:34-50`). The zero-bound and real 2,000 ms discriminator prove refusal, waiter removal, and later pool reuse (`webv0/packages/persistence/test/transactionDeadline.test.ts:23-27,73-103`).

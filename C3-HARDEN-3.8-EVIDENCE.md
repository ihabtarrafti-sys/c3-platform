## Cost envelopes

- **Owned-row availability:** the default idle-progress park while a row lock is held is **120,000 ms**. Each valid list page, completed prefix listing, or completed delete restarts that idle clock; malformed/no-progress storage work is aborted/raced and, when PostgreSQL cleanup progresses, the lock is released. This is an idle-storage bound, not a total row/pass or database-cleanup bound (`webv0/apps/api/src/erasureJanitor.ts:14-18,54-96,154-215,245-278`; `webv0/apps/api/src/storage.ts:90-115,144-175`).
- **Boot readiness:** pre-listen waiting is **30,000 ms by default**, configurable only to a positive value no greater than **300,000 ms**. Reaching the budget does not cancel or weaken the boot safety pass; it continues under observation while the interval is armed (`webv0/apps/api/src/erasureJanitor.ts:17-18,348-430`; `webv0/apps/api/src/env.ts:76-82`; `webv0/apps/api/src/server.ts:35-53`). The bound assumes the event loop can schedule the timer; no wall-clock promise is made across process suspension or event-loop starvation.
- **Owner availability:** an owner call during the same process's active owner pass returns an immediate rerun-required result; a cross-instance locked row returns `incomplete:true`; an owner call during an interval waits for that active pass plus one fresh owner pass (`webv0/apps/api/src/erasureJanitor.ts:294-340`; `webv0/apps/api/test/erasureJanitor.test.ts:275-357,482-530`). That queued path has no fixed total duration because it includes the remaining active pass and a progressing fresh pass.
- **No numeric byte-lifetime ceiling:** total pass duration scales with permanent-row count, storage pages/keys, successful progress, database work, and outages. The 24-hour maximum interval, 120-second idle-progress timeout, and readiness budget do not convert provider publication or infrastructure recovery into a fixed destruction deadline (`webv0/apps/api/src/erasureJanitor.ts:14-18,108-278,343-430`; `C:\Projects\C3-RESIDUAL-RISK-REGISTER.md:45-60`).

## The Question This Sprint Answers

*Can a Mission carry a planning sequence that generates work before things become emergencies?*

Not: can we record that flights are booked.  
Not: can we show a budget.  
The question is: can the platform tell an operator what planning work is due — before it's missed?

Milestones are the planning spine of Mission v2. Without them, travel records are a logistics table and budgets are a financial sheet. With them, the entire planning layer becomes operational: sequenced, owned, deadline-driven, and connected to the work queue.

---

## Design Principles for v1

**Milestones are planning checkpoints, not task managers.** A milestone marks that something has been arranged or confirmed — not the process of arranging it. "Flights booked" is a milestone. The back-and-forth with the travel agent is not.

**Status is computed, not stored.** Milestone status is derived from `PlannedDate` and `CompletedDate`, not written to a database. Same pattern as `OperationalGap` and `WorkItem`.

**The only write operation in v1 is "Mark Complete."** No creation, no editing, no deletion from the UI. Milestones come from mock seeds in Sprint 12. Editing is deferred. This keeps the write surface minimal while proving the planning loop end-to-end.

**Milestones that are Overdue or DueSoon generate WorkItems.** This is the primary value: the gap between "planned date has passed" and "mission departs in N days" becomes operational pressure in the Command Center — using exactly the same mechanism as credential gaps.

## Milestone Data Model

### `MilestoneStatus`

Computed from `PlannedDate` + `CompletedDate`. Never stored directly.

```
Complete   — CompletedDate is set. No further action required.
Overdue    — PlannedDate has passed. CompletedDate is null.
DueSoon    — PlannedDate is within 7 days. CompletedDate is null.
Upcoming   — PlannedDate is more than 7 days out. CompletedDate is null.
Blocked    — One or more dependency milestones are not yet Complete.
             (Modelled in v1 for future use. Generation skips Blocked milestones.)
```

The `DueSoon` threshold of 7 days is the v1 default. It may become configurable per milestone category in a future sprint.

### `MilestoneCategory`

The planning domain this milestone belongs to. Determines grouping and icon in the UI.

```
Roster      — Participant selection, eligibility confirmation
Compliance  — Visa applications, tournament registration, credential checks
Logistics   — Flights, accommodation, equipment shipping, freight
Finance     — Budget approval, expense planning
Documents   — Document collection and verification
Event       — Arrival, practice sessions, match day, departure
PostMission — Expense claims, prize payout, debrief, settlement
```

### `MissionMilestone` interface

```typescript
interface MissionMilestone {
  MilestoneID: string;           // Stable identifier, e.g. "ml-TR/2026/006-003"
  MissionID:   string;           // Foreign key to Mission
  Name:        string;           // Human-readable label, e.g. "Visa applications submitted"
  Description?: string;          // Optional supporting detail
  Category:    MilestoneCategory;
  Owner?:      string;           // Person or role responsible
  PlannedDate: string;           // ISO date — when this must be done
  CompletedDate?: string;        // ISO date — when it was done; null if open
  DependsOn?:  string[];         // MilestoneIDs that must complete first (modelled, not enforced in v1)
  Notes?:      string;           // Free-text coordinator notes
  CreatedAt:   string;           // ISO timestamp
}
```

### Computed view type

```typescript
interface MissionMilestoneView extends MissionMilestone {
  status:     MilestoneStatus;
  daysUntilDue: number | null;
  // positive = days remaining
  // negative = days overdue
  // null = already complete
}
```

### Deterministic WorkItem IDs for milestones

```
ml-{MilestoneID}
Example: ml-TR/2026/006-003
```

Same condition → same ID. Persistence-ready.

## Status Computation Rules

```
if CompletedDate is set:
  → Complete

else if any DependsOn milestone is not Complete:
  → Blocked   (v1: generated by computeMilestoneStatus; WorkItem generation skips these)

else:
  daysUntilDue = floor((PlannedDate - today) / 86400 seconds)

  if daysUntilDue < 0:
    → Overdue

  if daysUntilDue ≤ 7:
    → DueSoon

  else:
    → Upcoming
```

## WorkItem Connection

### New `WorkItemCategory`

```typescript
| 'MilestoneAlert'
```

`MilestoneAlert` items are generated when a milestone is `Overdue` or `DueSoon`. `Upcoming` and `Blocked` milestones do not generate WorkItems — there is no actionable pressure yet.

### New `WorkItemTrigger` variant

```typescript
| {
    type:               'MilestoneGap';
    missionId:          string;
    missionName:        string;
    milestoneId:        string;
    milestoneName:      string;
    daysUntilDue:       number;   // negative = overdue, positive = days remaining
    daysUntilDeparture: number;
  }
```

### Priority rules for `MilestoneAlert`

Added to `workItemPriority.ts` (first-match, at the top of the function after MissionDeparturePressure):

```
MilestoneAlert:

  if daysUntilDue < 0 AND daysUntilDeparture ≤ 7:
    → Immediate

  if daysUntilDue < 0:
    → High   (overdue milestone is always High, regardless of departure proximity)

  if daysUntilDue ≤ 3 AND daysUntilDeparture ≤ 14:
    → High

  else:
    → Normal
```

The reasoning: an overdue milestone is a planning failure that has already occurred. It warrants High attention unconditionally, because the window to act without consequence is shrinking. Only Immediate if the mission is also departing imminently (compounding pressure). Due-soon milestones are Normal unless departure is also imminent.

### WorkItem generation for milestones

New `generateMilestoneWorkItems(missions, allMilestones)` function added to `workItemGenerators.ts`.

For each ADR-002-eligible mission:
- Collect its milestones from `allMilestones`
- Filter to `status === 'Overdue' || status === 'DueSoon'`
- For each, emit one `MilestoneAlert` WorkItem

```typescript
WorkItem {
  id:             `ml-${milestone.MilestoneID}`,
  category:       'MilestoneAlert',
  title:          milestone.status === 'Overdue'
                    ? `${milestone.Name} — overdue`
                    : `${milestone.Name} — due in ${milestone.daysUntilDue}d`,
  detail:         `${mission.Name} · ${milestone.Category}${milestone.Owner ? ` · ${milestone.Owner}` : ''}`,
  owner:          milestone.Owner,
  ownerSource:    milestone.Owner ? 'ProtocolDefault' : 'Unrouted',
  dueDate:        milestone.PlannedDate,
  blockingMission: mission.Name,
  status:         'Open',
  trigger:        { type: 'MilestoneGap', ... },
  links:          { missionId: mission.MissionID },
  priority:       computeWorkItemPriority(...),
}
```

### Navigation from `MilestoneAlert` WorkItems

Same as `MissionDeparturePressure`: "View Mission" → Situation Room with `missionId` pre-selected.

The Situation Room already has mission pre-selection wiring from Sprint 11. The milestone section added in Sprint 12 will be visible when the user arrives.

## Service and Hook Architecture

### `IMilestoneService`

Follows ADR-001 parallel factory pattern. Two read methods + one write method:

```typescript
interface IMilestoneService {
  // Returns all milestones for a given mission, ordered by PlannedDate ascending.
  listMissionMilestones(missionId: string): Promise<MissionMilestone[]>;

  // Returns milestones for all missions in a single batch call.
  // Used by useWorkItems to avoid N+1 queries.
  listAllMilestones(): Promise<MissionMilestone[]>;

  // Marks a milestone complete. Sets CompletedDate to today.
  // Throws if milestone not found or already complete.
  completeMilestone(milestoneId: string): Promise<MissionMilestone>;
}
```

### Hooks

**`useMissionMilestones(missionId: string)`**
- Empty string → `enabled: false` (same gate pattern as `useMissionGaps`)
- Returns `{ milestones: MissionMilestoneView[]; isLoading: boolean }`
- Used by the Situation Room milestone section

**`useAllMilestones()`**
- Fetches all milestones in one call via `listAllMilestones()`
- Used by `useWorkItems` for Command Center generation
- Cache key: `queryKeys.milestone.all()`

**`useMarkMilestoneComplete()`**
- Mutation hook following the `useAddCredential` pattern
- On success: invalidates `queryKeys.milestone.all()` and `queryKeys.milestone.forMission(missionId)`
- Success toast: "Milestone marked complete"
- Error toast on failure, panel stays interactive

### Query key namespace

```typescript
queryKeys.milestone.all()
queryKeys.milestone.forMission(missionId: string)
```

Cache coherence: when `useMarkMilestoneComplete` succeeds, both keys invalidate. `useWorkItems` recomputes on the next render cycle — the completed milestone's WorkItem disappears from the Command Center.

## UI — Where Milestones Appear

### 1. Situation Room — Mission scope

When a mission is selected in the Situation Room scope selector, the milestone section appears below `MissionContextHeader` and above the gap list.

**`MissionContextHeader` extension (minimal):**
Add a milestone summary line: "N milestones · N overdue" with a status-coloured indicator. Clicking it scrolls to the milestone section (v1: does nothing; the section is always visible).

**`MilestoneSection` component:**

```
┌─────────────────────────────────────────────────────────┐
│  Planning Milestones                                     │
├──────┬──────────────────────────────┬──────────┬────────┤
│  ●   │  Roster confirmed             │  Mar 1   │  ✓    │
│  ●   │  Tournament reg. submitted    │  Mar 5   │  ✓    │
│  ○   │  Visa applications submitted  │  3d ago  │  Mark │  ← Overdue
│  ◐   │  Flights booked               │  In 2d   │  Mark │  ← DueSoon
│  ○   │  Accommodation confirmed      │  In 6d   │       │  ← Upcoming
│  ○   │  Equipment shipped            │  In 10d  │       │  ← Upcoming
└──────┴──────────────────────────────┴──────────┴────────┘
```

Status indicators:
- `●` filled circle = Complete (green)
- `○` empty circle = Upcoming (gray)
- `◐` half-filled = DueSoon (amber)
- `⊗` overdue marker = Overdue (red)

Date display:
- Complete: formatted date (e.g. "Mar 1")
- Overdue: "N days ago" (red)
- DueSoon: "In Nd" (amber)
- Upcoming: "In Nd" (gray)

"Mark" button:
- Appears only for Overdue and DueSoon milestones
- One-click: triggers `useMarkMilestoneComplete`
- Replaced with checkmark on success (optimistic UI or invalidation-driven)

Owner column: displayed if any milestone has an owner set. Hidden otherwise (no empty column).

### 2. Command Center — work queue

`MilestoneAlert` WorkItems appear in priority bands alongside credential and journey WorkItems, using `WorkItemCard`. No new UI component needed — the card renders the milestone data through the existing layout.

The "View Mission" action navigates to Situation Room with the mission pre-selected, where the operator sees the milestone section and can mark the relevant milestone complete.

**What the end-to-end loop looks like:**

```
Command Center:
  HIGH
  "Visa applications submitted — overdue · TR/2026/006 · Compliance · Operations"
  [View Mission]
         ↓
  Situation Room (TR/2026/006 scope):
  Planning Milestones:
  ⊗  Visa applications submitted  3 days ago  [Mark Complete]
         ↓
  Operator clicks "Mark Complete"
         ↓
  WorkItem disappears from Command Center.
  Milestone shows ● Complete in Situation Room.
```

This is the full planning loop, closed.

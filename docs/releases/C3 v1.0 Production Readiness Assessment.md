# C3 Platform — Production Readiness Assessment

**Assessment Date:** 2026-06-28  
**Version:** 1.0.0-mock  
**Assessor:** Engineering

---

## Summary

C3 is **conditionally production-ready**. The application can be deployed to a SharePoint tenant today and will run correctly in mock data mode. Transitioning to live SharePoint data requires IT-provisioned access and a single configuration change. No architectural work is needed to unblock that transition.

Several non-blocking hardening items (error boundaries, accessibility, tests) are recommended before a live production rollout with real users and real data.

## Remaining Engineering Work

Items that can be completed without SharePoint access:

### High Priority (before live user rollout)

| Item | Effort | Description |
|---|---|---|
| Error boundaries | Medium | Wrap screen and panel renders. Prevent full-page crashes from data errors. |
| Accessibility audit | Medium | Keyboard navigation, ARIA roles, focus management in panels and drawers. |
| `flowBaseUrl` in AppConfig | Small | Add the Power Automate base URL to AppConfig so `flowService` receives a real URL. Required for any write operation to work in production. |
| SharePoint list schema validation | Small | Confirm all field names in `contractMapper.ts`, `amendmentMapper.ts`, `personMapper.ts` match the actual SharePoint lists. |

### Medium Priority (production quality)

| Item | Effort | Description |
|---|---|---|
| Unit tests — hooks | Medium | Test `useContracts`, `useContract`, `useCreateAmendment` etc. against mock service registry. |
| Unit tests — mappers | Small | Test `contractMapper`, `amendmentMapper`, `personMapper` against sample SP list items. |
| Integration test — SP adapter | Medium | Smoke test `SharePointContractService.listContracts` against a real list. |
| Bundle code splitting | Small | Lazy-load screen chunks to address 685KB bundle warning. |
| `flowBaseUrl` per-environment config | Small | Separate flow URLs for dev/staging/production environments. |

### Lower Priority (post-launch)

| Item | Effort | Description |
|---|---|---|
| Fast Refresh warnings | Trivial | Separate `useApp` export from `AppProvider` to satisfy the ESLint rule. |
| Offline / stale-while-revalidate | Large | Configure TanStack Query `staleTime` and `gcTime` for better perceived performance on slow SP. |
| Optimistic updates | Medium | For write operations — update the UI immediately, roll back on failure. |
| Real-time updates | Large | Polling or SignalR for multi-user scenarios. |

## Deployment Prerequisites

To deploy C3 to a SharePoint tenant and run in **mock mode**:

1. Build: `npm run build:c3` — produces `packages/c3/dist/`
2. Build SPFx package: `npm run build` in `packages/c3-spfx-host` — produces `.sppkg`
3. Upload `.sppkg` to SharePoint App Catalog
4. Set `dataSourceMode: 'mock'` in the web part's property pane (or default config)
5. Deploy web part to a page

To switch to **live SharePoint mode**:

1. Set `dataSourceMode: 'sharepoint'` in the web part configuration
2. Set `spSiteUrl` to the target SharePoint site URL
3. Confirm SharePoint lists are provisioned with the expected schema
4. Confirm `flowBaseUrl` is set in AppConfig (requires `flowBaseUrl` field — see Technical Debt Register)

## Risks and Assumptions

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| SharePoint list schema differs from mappers | Medium | High | Validate all field names against real lists before going live. Run in mock mode first. |
| Power Automate flow not available at go-live | Medium | Medium | Amendment creation degrades gracefully (stub remains). All read operations unaffected. |
| Bundle size causes performance issues on SP | Low | Medium | Code splitting via Vite `manualChunks` can be added in one sprint. |
| PnP.js version incompatibility with tenant | Low | High | Validate in tenant environment before full rollout. |
| SPFx context not available in test deployment | Low | Medium | SPFxAuthService falls back gracefully; test with a dev user override. |
| No automated tests means regressions are invisible | High | Medium | Add unit tests for mappers and hooks before live data integration. |

### Assumptions

- The SharePoint site already has the required lists provisioned (Contracts, People, Amendments, Users, Activities).
- Field names in those lists match what the mapper layer expects.
- The Power Automate flow for `createAmendment` will be provisioned before write operations go live.
- A dedicated SharePoint App Catalog site is available for deployment.
- The SPFx solution is deployed to the app catalog with tenant-wide or site-specific permissions as appropriate.

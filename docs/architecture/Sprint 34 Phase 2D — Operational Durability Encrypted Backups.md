# Sprint 34 Phase 2D — Operational Durability: Encrypted Staging Backups & Restore Certification

**Status:** implementation + hosted certification COMPLETE (first manual backup

Provider limitation that drove this design: **Railway Hobby provides no native
PostgreSQL backup controls / PITR** in the owner's hosted account. This is an
independent daily **logical** backup — explicitly NOT PITR.

## Backup architecture
Railway cron `c3-backup-cron` (image `webv0/apps/backup/Dockerfile`) at
`15 2 * * *`, one-shot. `c3_backup` (read-only) → `pg_dump -Fc -Z6` → SHA-256 →
**age** encrypt (public recipient) → private R2 bucket
`c3-web-v0-staging-backups` → verify → `status/latest-success.json`. Sunday
runs also copied to `weekly/`; `manual/` on demand. Lifecycle: daily 15d /
weekly 90d / manual 180d / status never. Details:
[`docs/runbooks/C3-Web-V0-backup-restore.md`](../runbooks/C3-Web-V0-backup-restore.md).

## c3_backup privilege evidence (hosted)
`rolsuper=false, rolbypassrls=true, rolcreatedb=false, rolcreaterole=false,
rolreplication=false, rolcanlogin=true`; the ONLY non-superuser BYPASSRLS
principal. Live checks: reads all tenants (BYPASSRLS — a complete logical
backup needs every tenant), write DENIED, DDL DENIED, GRANT ineffective;
absent from the API runtime (API still `c3_app`). BYPASSRLS is the single
documented, narrow backup exception (migration 0006).

## R2 privacy / token scope
Bucket private (Standard, WEUR, no public dev URL, no custom domain, no CORS).
Two bucket-scoped S3 tokens: **writer** (Object R+W) on the cron only;
**monitor** (Object Read-only) for the GitHub monitor. No account-wide creds;
no secret in source or logs.

## Limitations / follow-ups
- **Monitor secret placement:** the R2 monitor token was placed on the Railway
  cron service (harmless read-only, but wrong home). Owner must copy the four
  values into **GitHub Actions secrets** (`R2_ENDPOINT`, `R2_BUCKET`,
  `R2_MONITOR_ACCESS_KEY_ID`, `R2_MONITOR_SECRET_ACCESS_KEY`), **remove**
  `R2_MONITOR_ACCESS_KEY_ID` + `R2_MONITOR_SECRET_ACCESS_KEY` from Railway, and
  re-run the monitor workflow to confirm a `stale=false` log line (the earlier
  "green" run without GitHub secrets is a no-op false-green).
- Cron schedule set via the Railway dashboard (no CLI); the first automatic run
  fires at the next 02:15 UTC.
- Restore is logical (last snapshot), not PITR.

## Rollback / disable
Clear the Cron Schedule (or delete `c3-backup-cron`) to stop backups + cost;
delete the R2 bucket to stop storage. `c3_backup` + migration 0006 are harmless
read-only and may remain. Full procedure in the runbook.

### Sprint 7 — Credential Management (Add Only)

Sprint 7 is the first write surface for the Credential domain. Its goal was to make credentials visible, structured, and operationally actionable against the capability model introduced in Sprint 6G — without upload, OCR, edit, delete, deactivate, or supersession. Add only.

Two interaction modes drive the entire sprint: a general-purpose entry point from the Credentials section and a capability-driven entry point from the Readiness tab. The second mode is where the capability model becomes visible to the operator for the first time, without exposing the underlying architecture.

#### 7A — `useAddCredential` hook

`hooks/useAddCredential.ts` — parallel factory mutation hook, structurally identical to `useInitiateJourney`. Calls `credentialService.addCredential(input)` and on success invalidates `queryKeys.person.credentials(input.HolderPersonID)`. The invalidation propagates through two consumers simultaneously: `usePersonCredentials` (credentials list) and `usePersonReadiness` (memoized on credential data — recomputes protocol evaluation automatically).

#### 7B — `AddCredentialPanel` component

`components/shared/AddCredentialPanel.tsx` — OverlayDrawer form following the established `StartJourneyPanel` / `CreateAmendmentPanel` pattern.

Props: `personId`, `open`, `onDismiss`, `capabilityHint?: CredentialCapability`.

**Capability-driven mode** (opened from Readiness → Resolve):
- Panel title changes to `Add Credential — Resolves {capabilityLabel}` (e.g., "Add Credential — Resolves Right to Work")
- Credential Type dropdown renders two groups using `credentialTypesFor(capabilityHint)`:
  - "Recommended for this requirement" — types that satisfy the capability
  - "Other credential types" — all remaining types
- Operator is guided, not blocked. Edge-case types remain selectable.
- This is the first place the capability model is visible to operators without exposing the underlying architecture.

**General mode** (opened from Credentials section):
- Panel title: `Add Credential`
- Flat ordered list of all 18 credential types

Form fields: Credential Type (required), Reference Number (required), Expiry Date (optional, date input with "leave blank if no expiry" hint), Issue Date (optional), Issued By (optional), Sub-Type (optional), Notes (optional textarea). Submit button shows "Registering…" during mutation. Success toast names the credential type: "Emirates ID added successfully."

#### 7C — DataRow `action` slot

`components/ui/DataRow.tsx` — added optional `action?: ReactNode` prop, rendered end-aligned in the same auto column as `right`. The right section now renders when either `right` or `action` is provided. Backward-compatible: all existing DataRow usages unchanged.

#### 7D — ReadinessPanel `onResolveObligation`

`components/shared/ReadinessPanel.tsx` — added optional prop:

```ts
onResolveObligation?: (capability: CredentialCapability) => void
```

When provided, each obligation row that is AtRisk or Unsatisfied renders a subtle "Resolve" button via the DataRow `action` slot. The button calls `onResolveObligation(obligation.satisfiedByCapability)` — signalling the capability upward without the panel knowing about `AddCredentialPanel`. Satisfied obligations never show the Resolve button. When the prop is absent, the panel is fully read-only and unchanged from pre-Sprint 7.

#### 7E — PersonProfile wiring

`screens/PersonProfile.tsx` — wired both panels:

- New state: `credentialPanelOpen`, `resolveCapability: CredentialCapability | undefined`
- `handleResolveObligation(capability)` sets `resolveCapability` and opens the panel
- `handleCredentialPanelDismiss()` closes and clears `resolveCapability`
- Credentials SectionCard header now has an `"Add Credential"` action button (general mode)
- Credentials EmptyState now has an `"Add Credential"` primary action button (general mode)
- ReadinessPanel call now passes `onResolveObligation={handleResolveObligation}`
- `AddCredentialPanel` mounted outside tab tree — drawer renders correctly regardless of active tab, does not unmount mid-animation

Also extracted `CREDENTIAL_LABELS` from PersonProfile into a new `utils/credentialLabels.ts` shared utility, removing the duplication between PersonProfile and AddCredentialPanel. `utils/credentialLabels.ts` also exports `CREDENTIAL_TYPE_ORDER` (display-ordered list of all 18 types) and `CAPABILITY_LABELS` (human labels for all 9 capabilities).

**Validated interaction flows:**
- Add Credential from Credentials section (no hint) — full flat list, credential appears in list immediately
- Add Credential from Credentials EmptyState — same
- Resolve from Readiness (capability hint) — grouped dropdown, panel title reflects capability
- After registration: Readiness recomputes, obligation row transitions status (Unsatisfied → Satisfied, AtRisk → Satisfied), overall header updates, Resolve button disappears on satisfied obligations
- Satisfied obligations — no Resolve button at any point

### Sprint 6 — Operational Model Proof of Concept (Phases 6A–6D)

Sprint 6 was restructured before a line of code was written. Rather than proceeding with the originally planned hardening and write-operations track, the sprint was reframed around a single question: *what is the smallest implementation that proves the Operational Model in practice?*

This reframing followed the approval of two foundational documents — **The Geekay Operational Model** and the **C3 Conceptual Framework** — and a key architectural decision: that Journeys belong to Person and are initiated by operational decision, not by contract or document. ("Journeys are initiated by operational decisions, not by documents or contracts. The Person owns the Journey.")

The four phases implemented together constitute the first working proof of the Operational Model: credentials are now first-class entities, protocols generate typed obligations, readiness is computed from real entity state, and the product now visibly answers "Are we operationally ready?"

#### Phase 6A — Credential Entity

Introduced `Credential` as a first-class operational entity — not a field on Person, not a document attachment.

- `types/credentials.ts` — `Credential` interface, `CredentialType` union, `CreateCredentialInput`. A credential is holder-bound, span-valid, and permanently retained (superseded, never deleted).
- `services/interfaces/ICredentialService.ts` — `listCredentialsForPerson`, `getCredential`, `addCredential`, `deactivateCredential`.
- `ServiceRegistry` extended with `credentials: ICredentialService`.
- `MockCredentialService` — three persons with credentials that intentionally exercise all three readiness states: Satisfied (PER-0003), AtRisk (PER-0001, visa expiring ~11 days), Unsatisfied (PER-0002, no visa or Emirates ID).
- `SharePointCredentialService` — graceful stub.
- `useCredentialService` — parallel factory hook bypassing the frozen SPService monolith. This established the pattern for all new domain services: read `config.dataSourceMode`, instantiate the appropriate service.
- `usePersonCredentials` — TanStack Query read hook.
- `PersonProfile` — Credentials SectionCard added. Non-interactive `DataRow` — this fix also prompted making `DataRow.onClick` optional, stripping `role="button"`, `tabIndex`, pointer cursor, and hover effects when no click handler is provided. Read-only rows no longer advertise false interactivity.

#### Phase 6B — Protocol and Obligation Evaluation

Introduced the type layer and evaluation logic for the Operational Model's Protocol and Obligation concepts.

- `types/obligations.ts` — `ObligationStatus` (`Satisfied | AtRisk | Unsatisfied`), `Obligation`, `ObligationEvaluation`. Span semantics documented: obligations are span-covering, not point-in-time. Phase 6B approximates span as a 90-day forward window; full span alignment to Journey/assignment periods is deferred.
- `protocols/onboardingProtocol.ts` — Pure evaluation function `evaluateOnboardingObligations(personID, credentials)`. Three obligations: Valid Passport, UAE Visa, Emirates ID. Candidate selection picks the non-expired credential with the latest expiry; uses existing `computeDaysToExpiry` from `dateUtils` (no duplication). Protocols are code-defined; no configuration UI.
- `protocols/index.ts` — Barrel export.

No UI was added in this phase. The evaluation function is available for Phase 6D to consume.

#### Phase 6C — Person-Owned Onboarding Journey

Introduced the `OnboardingJourney` entity and its full service stack.

- `types/journeys.ts` — `JourneyStatus`, `OnboardingJourney`, `InitiateJourneyInput`. `ContractID` is present but optional and explicitly documented: the Contract adds Protocol layers to the Journey; it does not own it.
- `services/interfaces/IJourneyService.ts` — `getActiveOnboardingJourney`, `listOnboardingJourneys`, `initiateOnboardingJourney`, `completeOnboardingJourney`.
- `MockJourneyService` — three seed journeys matching credential state: PER-0001 Active (gap open), PER-0002 Active (gap open), PER-0003 Completed (all satisfied). Write methods mutate an in-memory store for panel testing readiness.
- `SharePointJourneyService` — graceful stub. Read methods return null/empty; write methods throw (they cannot no-op safely since callers expect a returned journey).
- `ServiceRegistry` extended with `journeys: IJourneyService`.
- `useJourneyService`, `useOnboardingJourney`, `usePersonJourneys` hooks added.

No UI was added in this phase.

#### Phase 6D — Readiness Signal

Introduced the `ReadinessPanel` component and the Readiness tab on PersonProfile. This is the first point at which the Operational Model becomes visible in the product.

- `components/shared/ReadinessPanel.tsx` — Reusable display component. Takes a pre-computed `ObligationEvaluation`; renders a colored status header (green/amber/red surface with protocol name, summary line, evaluation timestamp) and a non-interactive `DataRow` per obligation with a status badge. Domain-aware (imports `ObligationEvaluation`), not in `components/ui/`.
- `PersonProfile` — Tab bar added (Profile | Readiness). Readiness tab: if no journey exists → EmptyState; if journey exists → Journey SectionCard (status badge, initiation reason, by/date/completion) + Onboarding Obligations SectionCard with `ReadinessPanel`.
- Readiness is computed — never stored. `evaluateOnboardingObligations(personID, credentials)` is called directly in the render path. No readiness field persisted anywhere.
- Bug found and fixed: `useOnboardingJourney` returned only Active journeys, causing PER-0003 (Completed journey) to show "No journey." Fixed by introducing `usePersonJourneys` (calls `listOnboardingJourneys`); PersonProfile now derives: active first, otherwise most recent.
- Mock data replaced throughout: `mockData.ts` updated with production-quality person and contract records. `'Team 30'`, `'Game 2'`, `'Department 30'` removed. Real team names (`GKE Fortnite`, `GKA Apex Legends`, `Creative`, `Operations`), departments (`Esports`, `Creative`, `Operations`), roles (`Performance Analyst`, `Head Coach`, `Player Operations Manager`), and nationalities applied. Placeholder names `'Guy Designer'` and `'Guy Editore'` replaced with `Sari Al-Khatib` and `Nadia Khoury`. Duplicate `PersonnelCode: 'AL/CH/001'` on PER-0009/PER-0010 corrected.

### Sprint 6E — Structural Freeze

Sprint 6E was a deliberate architectural consolidation sprint before any further feature work. Its goal was to freeze the structural foundations of the Operational Model so that future sprints build on a stable, coherent base rather than accumulating structural debt.

The sprint followed a formal architecture review conducted as a "new Principal Architect joining the project" exercise. Fourteen architectural issues were identified across four categories: inconsistencies, painful abstractions, overlooked concepts, and emergent capabilities. The five-phase freeze addressed the highest-priority findings.

#### 6E-1 — Journey Generalization

Renamed `OnboardingJourney` to the generic `Journey` with a `Type: JourneyType` discriminator. `JourneyType` covers `'Onboarding' | 'VisaRenewal' | 'TeamTransfer' | 'ContractRenewal' | 'Offboarding'`. All five types share a single entity, a single `IJourneyService`, and a single `useActiveJourney(personId, type)` hook. `useOnboardingJourney` becomes a thin wrapper. `suspendJourney` and `cancelJourney` added to the interface. `AssignedTo` added to `Journey` and `InitiateJourneyInput` — the first expression of the Ownership principle in the type system.

#### 6E-5 — Mission Interface Stub

Introduced `Mission` as a conceptual type in the type system: `MissionType`, `MissionStatus`, and the `Mission` entity with `ParticipantPersonIDs` and `Span: ObligationSpan`. Mission is the "activating intent" — the concept that selects Protocols and defines obligation spans. No service, no UI, no consumers. Present because it is the connecting node between `ProtocolContext` and real obligation spans.

#### 6E-2 — Obligation Span + Protocol Context

Introduced `ObligationSpan { from: string; to: string }` as the canonical span type (replacing the separate `MissionSpan` definition). `Obligation` gains `span?: ObligationSpan` (present when evaluation was span-aware) and `defaultOwner?: string` (sourced from protocol spec — the Ownership principle's second expression). `ProtocolContext { span?: ObligationSpan; mission?: Mission }` introduced; `evaluateOnboardingObligations` accepts it as an optional third parameter. Span resolution precedence: explicit span > mission.Span > 90-day default. All existing callers pass no context; behaviour is identical to pre-6E.

#### 6E-3 — usePersonReadiness Hook

Extracted protocol evaluation out of the PersonProfile render path into a dedicated `usePersonReadiness(personId, protocolFn, context?)` hook. The hook is protocol-agnostic — it accepts any function matching `(personId, credentials, context?) => ObligationEvaluation`. Evaluation is gated on credential load completion (returns `null` during loading, preventing the false-Unsatisfied result previously produced when credentials were still loading). Memoized on credential data. PersonProfile's Readiness tab now handles the loading/null/evaluation states explicitly.

#### 6E-4 — ADR-001: Service-Access Pattern

Produced `docs/adr/ADR-001-service-access-pattern.md`. Documents the two coexisting service access patterns: SPService (legacy compatibility layer, frozen, covers Contracts/People/Amendments/Users) and the parallel factory pattern (permanent standard, covers Credentials/Journeys and all future domains). Records the domain inventory, the alternatives considered (extend SPService, create SPService2 — both rejected), and the consequences.

**Sprint 6E build status:**
- build:c3 green after each phase
- lint:c3 green with pre-existing Fast Refresh warnings only
- No new UI, no visual changes to existing screens
- All existing PersonProfile behaviour preserved

### Sprint 6F — Start Onboarding Journey (First Operational Workflow)

Sprint 6F is the first validation sprint after the structural freeze. Its goal was to prove the Sprint 6E architecture through a complete end-to-end operational workflow: a real user action that triggers a service mutation, invalidates derived queries, and causes the UI to reflect new state — without any manual refresh.

**What this sprint proves:** The platform can move from passive evaluation to guided operational action. Journey initiation goes through the real service interface. The readiness panel that surfaces after creation is the real protocol evaluation. This is the first full loop: user action → service mutation → query invalidation → derived state update → UI reflects reality.

#### 6F-1 — useInitiateJourney

New mutation hook. Calls `journeyService.initiateJourney(input)` via the parallel factory pattern. On success, invalidates `journey.active(personId, type)` and `journey.list(personId)` using `input.PersonID` and `input.Type` from the mutation payload. No redundant personId parameter — all invalidation is driven from the input shape. `Notes?: string` added to `InitiateJourneyInput` (it was already on the `Journey` entity; the input type was not carrying it through).

#### 6F-2 — StartJourneyPanel

New `OverlayDrawer` form component in `components/shared/`. Follows the `CreateAmendmentPanel` pattern exactly. Three fields: Assigned Owner (required), Initiation Reason (required), Notes (optional). `Type: 'Onboarding'` and `InitiatedBy: currentUser.displayName` set automatically. Submit disabled until required fields are non-empty; shows "Starting…" during mutation. On success: toast + dismiss + form reset. On error: toast, panel stays open, form state preserved.

#### 6F-3 — PersonProfile Wiring

`journey` derivation scoped to Onboarding journeys (`journeys.filter(j => j.Type === 'Onboarding')`), making the Readiness tab semantically correct regardless of other journey types a person may have. `hasOnboardingJourney` guard derived from the same filtered set — the "Start Onboarding Journey" button is suppressed when any Onboarding journey exists (Active, Completed, Suspended, or Cancelled). `StartJourneyPanel` mounted outside the tab tree so the drawer does not unmount mid-animation if the user was on the Profile tab when they opened it.

**Button visibility:**
- PER-0001, PER-0002 (Active Onboarding): no button
- PER-0003 (Completed Onboarding): no button
- PER-0004 through PER-0010 (no Onboarding journey): button visible

**Known limitation — in-memory persistence only:**
Journey mutations are currently in-memory only. The workflow is fully validated end-to-end: the journey is created, queries invalidate, the Journey card and ReadinessPanel surface automatically, and the operator immediately sees which obligations are satisfied and which are missing. However, a full page refresh resets the created journey because the `MockJourneyService` store is not persisted. This is expected behaviour at this stage. Persistence is deferred until the SharePoint / Power Automate data layer is implemented.

### Sprint 6G — Capability-Based Credential Migration

Sprint 6G is a structural foundation sprint, completed before Credential Management UI was built. Its purpose was to correct the Credential model before it hardened any further: the previous `OnboardingProtocol` evaluated credentials by document name (`satisfiedByTypes: CredentialType[]`), which was implicitly UAE-centric and would have made C3 structurally provincial.

**Locked decision:** Protocols should ask for operational capabilities, not specific document names.

The three-layer model:

1. `CredentialType` — what the document **is** (Passport, Visa, Iqama, EmiratesID, …)
2. `CredentialCapability` — what the document **provides** (Identity, Travel, RightToWork, …)
3. Protocol obligation — what the operation **requires** (`satisfiedByCapability: CredentialCapability`)

This keeps the Operational Model jurisdiction-agnostic: the same `OnboardingProtocol` evaluates correctly whether a person holds a UAE Emirates ID, a Saudi Iqama, or a Labour Card from any other jurisdiction, because protocols express what they need (RightToWork) and the capability map resolves which document types satisfy that need.

#### 6G-1 — CredentialCapability Type

`CredentialCapability` added to `types/credentials.ts`:

```
Identity | Travel | RightToWork | RightToReside |
CompetitionEligibility | TransferEligibility |
HealthCoverage | HealthClearance | FinancialAccess
```

`CredentialType` expanded from 7 to 18 jurisdiction-neutral values, grouped by domain: Identity & Residency, Travel Authorisation, Work Authorisation, Competition & Transfers, Health, Financial.

#### 6G-2 — Obligation Interface Migration

`Obligation.satisfiedByTypes: CredentialType[]` replaced with `Obligation.satisfiedByCapability: CredentialCapability`. The field shift is significant: the Obligation now records what the protocol **required**, not which credential types happened to satisfy it. The former is a stable protocol concept; the latter is a jurisdiction-specific implementation detail.

#### 6G-3 — Credential Capability Map

`protocols/credentialCapabilities.ts` — single source of truth for the CredentialType → CredentialCapability resolution:

- `CREDENTIAL_CAPABILITIES: Record<CredentialType, CredentialCapability[]>`
- `credentialProvides(type, capability): boolean` — used by protocol evaluation filters
- `credentialTypesFor(capability): CredentialType[]` — available for future UI hints ("Which credentials would satisfy this gap?")

Key mapping decisions:
- **Passport → `['Identity']` only.** A passport proves who the person is; it does not by itself authorise entry into a specific jurisdiction. Travel readiness must come from a Visa, Entry Permit, or equivalent territorial authorisation credential. This is architecturally important: if Passport were given `['Travel']`, PER-0001's expiring Visa would be masked by the valid Passport (2032 expiry), destroying the AtRisk signal.
- **EmiratesID and Iqama both → `['Identity', 'RightToWork', 'RightToReside']`.** They are operationally equivalent — the same bundled residency/work credential in different jurisdictions.
- **ResidencePermit → `['RightToReside']` only.** Without jurisdiction context, a generic Residence Permit cannot be assumed to include work rights. Jurisdiction refinement is deferred.

#### 6G-4 — OnboardingProtocol Migration

`protocols/onboardingProtocol.ts` fully migrated:

- `ObligationSpec.satisfiedByTypes` removed; `satisfiedByCapability: CredentialCapability` added.
- Three obligations updated to capability-based, jurisdiction-neutral specs:

  | ID | Requirement | Capability |
  |---|---|---|
  | `identity` | Identity Document | `Identity` |
  | `right-to-work` | Right to Work Authorization | `RightToWork` |
  | `travel` | Travel Authorization | `Travel` |

- Evaluation filter changed from `spec.satisfiedByTypes.includes(c.Type)` to `credentialProvides(c.Type, spec.satisfiedByCapability)`.
- All candidate selection, span-aware AtRisk, and 90-day threshold logic is unchanged.

The obligation IDs also changed from `passport / uae-visa / emirates-id` to `identity / right-to-work / travel`. The old IDs named documents; the new ones name operational requirements.

**Behaviour verification (current mock data, no context, 90-day default):**

| Person | Before | After |
|---|---|---|
| PER-0001 (Abdulaziz) | AtRisk — Visa 11 days | AtRisk — Travel Authorization 11 days |
| PER-0002 (Mohammad) | Unsatisfied — no Visa, no Emirates ID | Unsatisfied — no Travel Authorization, no RightToWork |
| PER-0003 (Diab) | Satisfied | Satisfied |

Overall statuses identical. The specific obligation labels are now operational rather than document-specific.

- build:c3 green, 0 errors
- lint:c3 green, 0 errors (3 pre-existing Fast Refresh warnings only)
- No UI changes; all ReadinessPanel output preserved via updated `requirement` strings

- "Add Amendment" button introduced in ContractProfile's PageHeader actions.

**Known limitation — amendment persistence deferred:**
`flowService.createAmendment` is currently a stub that returns a resolved promise without persisting to any data store. The form validates, the mutation fires, the success toast appears, and the panel dismisses correctly. However, the created amendment does not appear in the Amendments count or register because no persistence layer exists yet. Persistence will be implemented in the future write/Power Automate integration phase when the real Power Automate flow URL is available.

### Sprint 11 — Command Center: Operational Work Queue ✅ COMPLETE

Sprint 11 replaced the Command Center's contract KPI dashboard with a shared operational work queue. This sprint asked and answered a different question from all prior sprints: not "what entity should we model?" but "how does C3 decide what deserves an operator's attention right now?"

The answer turned out not to be another entity. It was a new primary surface — sitting above the existing models — that translates computed operational state into actionable work.

The sprint began with a holistic review of the platform after Mission v1. The review identified that the platform's largest gap was not capability — it was the absence of an operational inbox. An operator arriving at C3 had to navigate to the Situation Room, filter by ownership state, and synthesize the queue themselves. The Command Center was not earning its name.

#### The evidence/intent distinction

The key architectural insight from the pre-implementation design phase:

**OperationalGap = evidence.** It describes what is operationally true. "This person has no valid Travel credential. The obligation is Unsatisfied. No one owns it."

**WorkItem = intent.** It describes what an operator needs to do. "Start a readiness Journey for Alex Chen. 3 unrouted gaps."

OperationalGap is not replaced by WorkItem. OperationalGap is one of the triggers that creates work. Future triggers — ContractExpiry, RosterChange, ManualEntry — will extend the queue without touching the existing gap model.

#### S11-1 — WorkItem type system

`types/workItems.ts` introduced the full WorkItem contract:

- `WorkItemCategory` (5 variants): `CredentialRenewal | CredentialAcquisition | JourneyInitiation | ObligationRouting | MissionDeparturePressure`
- `WorkItemPriority` (3 tiers): `Immediate | High | Normal` — distinct from `UrgencyTier (Critical | High | Medium)`
- `WorkItemStatus`: `Open | InProgress | Resolved` — `Open` always in Sprint 11; `InProgress` and `Resolved` modelled for the persistence path
- `OwnerSource` (4 values): `ObligationAssignment | JourneyOwner | ProtocolDefault | Unrouted` — drives owner badge confidence colour
- `WorkItemTrigger` union: `OperationalGap` (person-scoped) vs `MissionDeparture` (cross-person)
- `WorkItem` interface with deterministic ID strategy documented inline

Deterministic IDs per category:
- `ji-{personId}` — JourneyInitiation
- `ca-{personId}-{capabilitySlug}` — CredentialAcquisition
- `cr-{personId}-{capabilitySlug}` — CredentialRenewal
- `or-{personId}-{capabilitySlug}` — ObligationRouting
- `mdp-{missionId}` — MissionDeparturePressure

`capabilitySlug` is `CredentialCapability` converted to kebab-case: `'RightToWork' → 'right-to-work'`.

The ID strategy ensures Sprint 11's computed WorkItems are persistence-ready without any schema changes when a WorkItem store is introduced.

#### S11-2 — Generation engine and priority computation

Two pure utility files. No React, no hooks, no side effects.

`utils/workItemPriority.ts` — `computeWorkItemPriority(category, trigger, blockingMission, daysUntilBlockingMission)`. First-match rules:

- `MissionDeparturePressure`: driven entirely by departure proximity. ≤ 7 days → Immediate. 8–30 days → High. All MDP items are within the 30-day window, so minimum is High.
- `ObligationRouting`: caps at High (routing is an organisational step; credential work takes Immediate). Without mission pressure: always Normal.
- Person-scoped items with mission pressure: ≤ 7 days → Immediate; ≤ 14 days → High.
- Gap-urgency-driven: `CredentialAcquisition` and `CredentialRenewal` with Critical or High urgency → High. `JourneyInitiation` with Critical → High.

`utils/workItemGenerators.ts` — `generateWorkItems(gaps, missions) → WorkItem[]`. Five-step pipeline:

1. Build mission pressure index: filter to ADR-002-eligible missions within 30-day departure window. Active missions → days = 0 (happening now). PostMission excluded. Build `personMinDeparture: Map<personId, number>` and `personBlockingMission: Map<personId, string>`.
2. Group all gaps by personId.
3. Per person: partition by `ownershipState`. `Unrouted` → JourneyInitiation (one per person). `Routed` → ObligationRouting (one per capability type). `Covered` → CredentialAcquisition or CredentialRenewal (one per capability type; discriminated by `daysToExpiry === null`).
4. Per upcoming mission: `MissionDeparturePressure` item if any participant has open gaps.
5. Sort: priority band → MDP first within band → items with blockingMission before items without → dueDate ascending.

Design note (recorded): The `daysToExpiry === null` discrimination for CredentialAcquisition vs CredentialRenewal is correct for the current `OperationalGap` shape. Long-term, this distinction should come from explicit obligation status or trigger metadata when `OperationalGap` carries `ObligationStatus` directly.

#### S11-3 — `useWorkItems` hook

`hooks/useWorkItems.ts` — thin composition hook. Wires `useOperationalGaps` and `useMissions` into `generateWorkItems`. No new fetches — both hooks are already in the TanStack cache from the Situation Room. Two `useMemo` calls: one for items (recomputes on data change), one for counts (single-pass reduce). Stable empty return during loading.

Returns `{ items: WorkItem[]; counts: WorkItemCounts; isLoading: boolean }`.

#### S11-4 — Command Center rewrite + navigation wiring

Four additional changes:

`types/screens.ts` — `situation-room` screen variant extended with `missionId?: string`. `SituationRoom` now accepts `initialMissionId?: string` prop and initialises `selectedMissionId` state from it. `AppShell` passes `screen.missionId` through. This makes `MissionDeparturePressure` navigation exact: clicking "View Mission" in the Command Center pre-selects the correct mission scope in the Situation Room.

`components/shared/WorkItemCard.tsx` — new shared component. Priority dot (colour-coded: critical/warning/gray), title + detail text, owner badge (colour-coded by OwnerSource confidence level), mission chip (brand-accented), due date chip (red when ≤ 7 days), action button with category-specific label.

`screens/CommandCenter.tsx` — full rewrite. Three priority bands (Immediate, High, Normal) rendered as Fluent UI-styled panels. Empty bands are suppressed — a queue with only Normal items shows no Immediate or High section. Immediate band carries a critical-tint header to draw the eye. - All other items → `navigate({ id: 'person-profile', personId: item.links.personId, tab: 'readiness' })`

### Phase 1 — Documentation and Audit

#### S14-5 — GapFilter Audit

**Finding corrected during this task:** The initial Sprint 14 proposal assumed GapFilter fields were dead/unused. The audit revealed all three fields (personIds, context, protocols) ARE internally handled by `useOperationalGaps` — they are read and acted on in the hook. The only accurate statement was that no UI consumer currently passes a non-null filter.

**Action taken:** Comment-only update. No code changes.

- `types/situation.ts` — Updated GapFilter JSDoc from misleading "Sprint 8: filter is unused" to accurate description: all three fields are wired; each awaits a UI consumer
- `hooks/useOperationalGaps.ts` — Updated stale comment from "Sprint 8" to "S14-5 audit (Sprint 14)"
- Added `ProtocolContext.mission` is targeted for removal in S14-4 (forward notice)

**Key lesson:** The documentation was wrong. The code was right. Fixing the documentation, not the code, was the correct outcome.

#### S14-6 — Journey ADR (ADR-003)

`docs/adr/ADR-003-journey-definition.md` written and accepted.

Seven questions answered explicitly:
1. A Journey is an operational engagement accountability record — not a task tracker
2. Journey lifecycle: Active → Completed / Suspended / Cancelled. No automatic state advancement
3. Multiple concurrent Journey types permitted; multiple of the same type are not
4. MissionID on Journey is informational only — does not scope the Journey to a Mission
5. Covered requires an Active Journey with a matching `obligationAssignment` entry
6. `Journey.AssignedTo` = governance owner; `ObligationAssignment.assignedTo` = execution owner
7. ObligationAssignments should be stored as a separate SharePoint list (JSON column acceptable for v1 simplification)

**Open questions flagged for operator validation:**
- A. Does ops need `addObligationAssignment` post-Journey-initiation?
- B. Multi-type Journey type precedence for obligation matching
- C. Suspend vs Cancel operational semantics
- D. Maximum concurrent Journey count per person in practice

#### S14-7 — SharePoint Integration Risk Assessment

`docs/architecture/SharePoint Integration Risk Assessment.md` written and accepted.

**Critical risk identified:** `obligationAssignments` has no native SharePoint equivalent. Covered ownership state is impossible in production until the obligation assignments storage model is implemented (either a separate JourneyObligationAssignments list, or JSON in a multi-line text column).

**High risks identified:** date format (ExpiryDate), CredentialType string matching, Status/Type/Category/Direction choice field string matching, ParticipantPersonIDs (no SP array type — now resolved in S14-2), `FinanceLineCategory` exhaustive Record (runtime error on unknown SP values).

**Integration sequence recommended:** Credentials → Journeys → Missions + Participants → Milestones → Finance.

### Phase 2 — Structural Refactors

#### S14-1 — Extract computeGapsForPeople

**Problem:** The ownership-state algorithm (Unrouted/Routed/Covered resolution) and obligation-to-gap conversion were duplicated identically in `useOperationalGaps` and `useMissionGaps`. A bug in the three-state resolution block would require fixes in two places simultaneously.

**Solution:** New pure utility `utils/gapComputation.ts` — no React, no hooks, no side effects.

**Exports:**
- `PersonInfo` — normalised person descriptor accepted by the function
- `ComputeGapsOptions` — optional mission scope (missionId, missionName, missionEndDate)
- `computeGapsForPeople(people, credentialsByPerson, journeyByPerson, protocols, context?, options?)` → `OperationalGap[]`

**Key design:** Each hook normalises its source type to `PersonInfo[]` before calling. `useOperationalGaps` derives from `Person[]`; `useMissionGaps` derives from `MissionParticipant[]` + People Map lookup. The function is agnostic to the source type. Both hooks become orchestration layers only: fetch, map, compute, return.

**Files modified:** `hooks/useOperationalGaps.ts`, `hooks/useMissionGaps.ts`
**Files created:** `utils/gapComputation.ts`

**Build:** Exit 0, no regressions.

#### S14-4 — Narrow ProtocolContext

**Problem:** `ProtocolContext.mission?: Mission` caused protocol files to import from the Mission domain. Protocols only needed `mission.Span` — they had no business receiving the full Mission entity.

**Solution:** Remove `mission?: Mission` from `ProtocolContext`. Add `jurisdiction?: string` for future jurisdiction-aware protocols.

**Key finding:** The `mission?.Span` fallback in `resolveSpan` was already unreachable — `useMissionGaps` had always set `context.span` explicitly. The removal was safe without any behaviour change.

**Files modified:**
- `types/protocols.ts` — removed Mission import + `mission?` field; added `jurisdiction?`
- `protocols/onboardingProtocol.ts` — simplified `resolveSpan`: removed unreachable `context.mission?.Span` branch

`protocols.ts` now has zero dependency on the Mission domain type.

**Build:** Exit 0, no regressions.

#### S14-2 — Resolve Mission Participant Representation

**Audit finding:** `Mission.ParticipantPersonIDs: string[]` and `MissionParticipant.PersonID` were two sources of truth for the same fact, kept in sync manually. `ParticipantPersonIDs` would require a multi-value field at SharePoint integration time — already a Critical risk in S14-7.

**Three call sites identified:**
1. `SituationRoom.tsx:768` — participant count display
2. `workItemGenerators.ts:377` — `generateMissionDeparturePressure` gap iteration
3. `workItemGenerators.ts:564` — `generatePresencePressureWorkItems` departure pressure index

**Decision:** Remove `ParticipantPersonIDs` from `Mission` entirely. `MissionParticipant.PersonID` becomes the authoritative source.

**Files created:**
- `hooks/useAllMissionParticipants.ts` — fetches all MissionParticipant records; derives `participantPersonIdsByMission: Map<string, string[]>`

**Files modified:**
- `types/mission.ts` — removed `ParticipantPersonIDs: string[]`
- `services/interfaces/IMissionService.ts` — added `listAllMissionParticipants(): Promise<MissionParticipant[]>`
- `services/mock/MockMissionService.ts` — removed field from both mock records; added `listAllMissionParticipants()` returning all MOCK_PARTICIPANTS
- `services/sharepoint/SharePointMissionService.ts` — added graceful stub
- `hooks/queryKeys.ts` — added `mission.allParticipants()` key
- `utils/workItemGenerators.ts` — `generateWorkItems` gains 4th parameter `participantPersonIdsByMission`; both generator functions iterate from the map
- `hooks/useWorkItems.ts` — wires `useAllMissionParticipants`; passes map to `generateWorkItems`
- `screens/SituationRoom.tsx` — participant count now from `useMissionParticipants(selectedMission?.MissionID ?? '')`; hook shares TanStack Query cache key with `useMissionGaps` (no extra network call)

**Key note:** `MissionFinanceLine.ParticipantID` was already a PersonID (confirmed in mock data). Finance domain unaffected.

**Build:** Exit 0, no regressions.

#### S14-3 — Split workItemGenerators.ts

**Problem:** `workItemGenerators.ts` was a 628-line monolith combining constants, sort helpers, ID helpers, three per-person generators, one mission generator, one milestone generator, and the pipeline entry point. Finding and auditing any single generator required scanning the entire file.

**Solution:** Split into a directory with five focused modules.

**Directory structure created:** `utils/workItemGenerators/`
- `helpers.ts` — constants (`DEPARTURE_PRESSURE_WINDOW_DAYS`, `URGENCY_ORDER`, `PRIORITY_ORDER`) + 7 shared pure functions (`toCapabilitySlug`, `pickMostUrgent`, `groupByCapability`, `resolveOwnerSource`, `isoDaysFromToday`, `getDaysUntilDeparture`, `sortWorkItems`)
- `gapGenerators.ts` — `generateJourneyInitiation`, `generateObligationRouting`, `generateCredentialItems`
- `missionGenerators.ts` — `generateMissionDeparturePressure`
- `milestoneGenerators.ts` — `generateMilestoneWorkItems`
- `index.ts` — `generateWorkItems` entry point + 6-step pipeline

**Import path preservation:** `utils/workItemGenerators.ts` converted to a zero-logic re-export barrel (`export { generateWorkItems } from './workItemGenerators/index'`). All existing consumers import from `@c3/utils/workItemGenerators` unchanged.

**Implementation exception:** The original acceptance criteria stated the monolithic `.ts` file should no longer exist. Deletion of `utils/workItemGenerators.ts` was blocked by the Windows mount's filesystem permissions (`rm: Operation not permitted`). The barrel file approach is functionally equivalent and architecturally correct. The `.ts` file contains zero logic. Documented here for Sprint 14 closeout.

**Invariants confirmed:**
- All 6 deterministic ID formats preserved (`ji-`, `or-`, `ca-`, `cr-`, `mdp-`, `ml-`)
- Sort order preserved: priority band → MDP first → blockingMission → dueDate
- Pipeline step order preserved: 1→2→3→4→5→6
- No WorkItem logic, priority rules, or output ordering changed

**Build:** Exit 0, no regressions.

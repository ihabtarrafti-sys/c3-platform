## Appendix — C3 Comms tech notes (owner briefing, 2026-07-10; re-quote pricing at spec time)

- **WebRTC, not SIP.** SIP/OpenSIPS/Asterisk = telephony world (PBX, phone numbers, carrier trunks) — wrong tool for browser-first staff comms; never self-run. PSTN dial-in, if ever demanded, = a Twilio SIP trunk integration on top.
- **"Free" = license, never operations.** Self-hosting media infra costs servers + bandwidth + engineering attention (the scarcest C3 resource). Hence: buy infra, build product.
- **Cost shape (ballpark):** chat = in-house Postgres + WebSockets, ~free at tenant scale (and 80% of module value). 1:1 calls = WebRTC P2P + small TURN relay (~$10–20/mo VPS). Group calls = SFU required; managed (LiveKit Cloud / Daily-class) ≈ $0.003–0.007 per participant-minute → 5-person 30-min call ≈ $0.50–1; org-month ≈ $50–100, inside module pricing; free tiers cover early usage.
- **No-lock-in path: LiveKit** — open source AND managed with the same API. **Owner direction (2026-07-10): build-it-ourselves preferred — "same as C3"** = own the product, stand on open infra (the Postgres analogy: we didn't write Postgres/Fastify/React either). Layer answers: chat = write 100% ourselves; 1:1 calls = write ~90% (signaling on our WebSockets; browsers carry the media stack; run coturn); group calls = write the product, RUN the open-source engine (self-hosted LiveKit OSS on our servers, no per-minute fees), never REWRITE the media engine (SFU internals = solved specialized problem). **Plan: prototype SELF-HOSTED from day one of the module** (single-VPS LiveKit is manageable at our scale); managed cloud = pressure-release valve (same API) if ops burden bites. Deciding factor stated honestly: who debugs call quality at 2am.
- **Build order inside the module:** Phase 1 chat (in-house, cheap) → Phase 2 1:1 calls (P2P + TURN) → Phase 3 group calls (self-hosted LiveKit SFU, managed as fallback) → PSTN only on paid demand.
